## Reconciling the Grok ↔ Codex disagreement on regime filter

Grok (S7): ship pure OBI only in v1; add regime in v1.1 based on observed losing trades.
Codex (Q5): ship OBI + regime in v1; regime is cheap protection against chop.

**Resolution:** ship a **binary hard-coded regime gate** in v1, not a tunable regime classifier.

- Concrete rule: compute the v96 choppiness ratio; if `choppiness > 0.65` → skip entry. Hard-coded threshold, no tuning.
- Not a tuning surface: one parameter, one fail mode, does not interact with OBI threshold calibration.
- Satisfies Codex (cheap chop protection) without adding Grok's concern (second tuning dimension blocking OBI learning).
- Threshold moves from hard-coded to config-driven in v1.1 once recorded data shows the right value.

## Where Codex's review shifts the plan (must adopt)

### C-S1 — Build Bot E0 recorder before any Bot E trader code

**The biggest structural change in the entire plan.** Previously (Grok-revised): phase 0 = latency logger + extended backtester. Codex's approach is cleaner:

**Phase 0 (revised) = Bot E0 recorder:**
- Subscribes to Polymarket WSS: CLOB book channel + orders_matched channel, for BTC/ETH/SOL 15-min markets
- Subscribes to Binance WSS (BTC/ETH/SOL trades)
- Subscribes to Chainlink (Data Streams if accessible, else on-chain + CEX fallbacks)
- **Captures everything with LOCAL receipt timestamps** (critical — measures our infra, not someone else's)
- Writes to a dedicated SQLite (or parquet) with rich indexing
- **No order placement. Zero.**
- Runs 1–2 weeks minimum on the current LXC+VPN stack

**Phase 0.5 = Extended backtester (Grok's S1) then consumes Bot E0's data:**
- Add 2026 fee curve, slippage model, latency injection
- Replay recorded data with simulated entries at candidate OBI thresholds
- Output: conditional expectancy by (minute-to-expiry × liquidity bucket × regime)

**Phase 1 = Bot E v1 trader** — only if Phase 0.5 shows positive EV after all costs.

If Phase 0.5 shows no edge, we close Bot E entirely. This is the "save the 100 hours" path.

### C-S2 — Write ADR-022 before any code

Strategy pivot from near-resolution arb (the original approved Bot E scope) to OBI-directional is a material scope change and deserves explicit governance. Per CLAUDE.md: "Every material decision gets an ADR."

### C-S3 — Kelly fraction even lower for paper

When we eventually get past Phase 0.5 into Phase 1 trader code, start with **fixed tiny notional (e.g., $2/trade) or k=0.10–0.15**, not k=0.25. Only graduate to k=0.25+ after 300+ calibrated paper trades.

### C-S4 — Primary + secondary tagging schema

Update the reason_code design:
- **Primary tag** (`strategy_signal`): strategy-specific, what the bot actually matched. Examples: `obi_above_threshold`, `obi_below_threshold`, `regime_veto_choppy`, `whale_entry_copied`, `bucket_ladder_elimination`, `cex_lag_arb`, `near_resolution_99c`
- **Secondary tag** (`reason_code`): the five-question enum for cross-strategy attribution: `{price_wrong, liquidity_weak, linked_lag, repricing, ev_structure}`
- **Tertiary** (`reason_detail`): optional free text for exceptional cases; required when primary is `manual_override`

This satisfies Codex's "too philosophical, too lossy" critique without throwing away the five-question framing.

### C-S5 — Additional halts beyond Grok's list

- **Feed-divergence halt**: when two independent price sources (CEX vs Chainlink) disagree by >N bps → halt entries until reconvergence
- **Clock-sync halt**: if local clock drift >250ms vs NTP → halt (reconciliation becomes unreliable)
- **Execution-quality halt**: if rolling 50-trade average slippage > threshold → halt (book conditions have changed)
- **"Blind" halt**: if any critical feed silent for >N seconds → halt entries but keep lifecycle management running

### C-S6 — Assume public WSS unreliable (aligns with Grok G4/S5)

Both reviewers flagged this. No-gap recovery path is mandatory for Bot E0 recorder — a silent stall during capture silently corrupts calibration. Recorder writes a heartbeat row every N seconds; analysis script rejects any gap > threshold for that time range.

## Where I'd still push back (partial)

### R-C1 — Regime filter in v1

Agree with Codex on inclusion; want to constrain it tightly so Grok's concern is neutralised:
- **One threshold only** (choppiness > 0.65 → skip)
- **Hard-coded, not env-overridable, in v1**
- **Not a tuning surface during Phase 0.5 calibration** — OBI threshold is the only knob we tune on recorded data
- Promoted to env-configurable in v1.1 once data justifies a different value

### R-C2 — "Missing halts" list

Agree with additions. Feed-divergence, clock-sync, execution-quality halts are all low-cost additions. "Blind halt" is essentially an extension of our existing stale-feed halt.

Don't add all of them in v1. In Phase 0 recorder, just log the violations (don't halt — we're not trading). When we graduate to trader code, the violation logs tell us which halts actually fire and at what rates, so we tune thresholds empirically.
