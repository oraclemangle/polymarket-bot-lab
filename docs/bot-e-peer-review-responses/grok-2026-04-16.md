## Where Grok's review shifts the plan (must adopt)

### S1 — Add phase 0 "foundation" before Bot E v1 signal code

1. **Latency benchmark**: 24h passive logger measuring round-trip to `ws-subscriptions-clob.polymarket.com` + Binance WS + Chainlink fallback, from our current LXC+VPN setup. Fail threshold: 95th-percentile E2E >120ms → OBI is degraded from this infra; revisit eu-west-1 VPS.
2. **Backtester extension**: `core/backtest.py` currently has take-the-touch fill model. Needs (a) 2026 Polymarket fee curve (flat vs dynamic, verified against docs), (b) realistic slippage model derived from logged book depth, (c) latency injection (delay between signal fire and fill), (d) replay against historical orders_matched data (need to confirm availability or build a capture harness).

Phase 0 blocks any signal/executor code.

### S2 — Rewrite paper gate for directional strategy

Current spec: 200 trades, ≥75% WR. Grok's correct math: at p=0.625, P(≥75% on 200) ≈ 1–2%. The gate is statistically infeasible.

**Revised gate for Option B:**
- ≥300 trades
- Positive net EV after modelled fees + slippage (from backtester)
- Sharpe ≥ 1.0
- Max drawdown ≤ 25%
- Survived ≥ 2 distinct regimes without kill-switch

### S3 — Correlation cap on crypto bucket in v1

Our spec had aggregate 30% exposure cap. Grok: BTC/ETH/SOL at the same 15-min resolution are highly correlated; individual Kelly fails, one black-swan wipes 30–40%. Fix in v1 config:
- Option a: `CRYPTO_BUCKET_NOTIONAL_CAP_FRAC = 0.15` (hard cap across BTC+ETH+SOL combined)
- Option b: Treat whole crypto bucket as one Kelly position, size by worst-correlation worst-case

Going with option (a) for v1 simplicity; revisit once we see real simultaneous-position stats.

### S4 — Fix dimensional ambiguity in regime classifier

Before writing `regime.py`, verify actual units of Chainlink BTC ticks. If they're USD, `|dir_10m| > 30` is meaningless at $85k. Likely fix: `REGIME_TREND_DIR_10M_MIN_BPS` or `_MIN_PCT` expressed as basis-points or percentage of current price. Parameterise, don't hardcode 30.

### S5 — Telemetry + resilience on day 1

Not optional. Add to v1 scope:
- Per-source latency histograms (writes to SQLite every N minutes)
- Stale-feed auto-pause at 300ms (existing spec had 500ms; Grok suggests tighter)
- Position-vs-on-chain reconciliation every scan cycle
- Per-signal P&L attribution (once we have trade history)
- WSS heartbeat pings + REST book-snapshot fallback every 30s + exponential-backoff reconnect + max-reconnect-storm guard

### S6 — Upgrade Bot D dual-write log level (partial concession on Q16)

Grok's critique: silent try/except on the ADR-021 dual-write lets data gaps accumulate. Operator's preference: trades must not be blocked.

Middle ground: upgrade the log level from `warning` to `error`, add a daily-rate counter, and Telegram-alert if the failure rate exceeds 1% of entries. Still never re-raises. Delivers on both: no silent gaps, no blocked trades.

## Where I'd push back (partial / full disagreement)

### R1 — Q16 "best-effort try/except is wrong for any trading bot"

Partial disagreement. The dual-write is bolt-on analytics, not trading-critical. Operator preference memory: "constraints must not block profit." A hard-fail on a markets-table insert error would halt Bot D for a data-integrity concern that trading doesn't depend on. Resolution: S6 above (upgrade log + rate alert), keep try/except.

### R2 — "Expand reason_code enum to 7–8 or add free-text"

Full disagreement on free-text. The five-question enum is a forcing function; free-text defeats it. Enum extension is acceptable IF genuinely needed (microstructure, correlation-lead could be real additions), but adding free-text makes every trade insert an essay. Resolution: hold the line on enum-only; extend when a concrete 6th category has real trades demanding it.

### R3 — "Add a global daily-EV monitor"

Partial — agree the signal is useful, disagree with making it a third halt mechanism now. Daily-EV rolling attribution is cheap to compute; start as a Telegram info-signal before it becomes a halt trigger. We don't yet know what threshold to halt on. Build the metric; graduate to halt once 30 days of data show the right threshold.
